# Roadmap

最終更新: 2026-04-15

## 前提

- 当面は `公開 FFLogs ログのみ対応` を維持する
- 構成は `静的フロント + Netlify Functions + Supabase` を維持する
- 先に `壊れにくさ` と `運用確認のしやすさ` を上げ、その後に機能追加を行う

## 現在地

実装済みの主な範囲:

- 公開 FFLogs レポート 2 件の読み込み
- Kill 戦闘の選択とプレイヤー一覧取得
- プレイヤー比較タイムライン表示
- フェーズ表示、ボス詠唱、デバフ、簡易 DPS 推移
- 初回ガイドページ
- ビルド情報表示
- 利用イベント保存と簡易 analytics ダッシュボード

現状の開発上の特徴:

- `scripts/data/fflogs.js` は 467 行
- `scripts/app/bootstrap.js` は 551 行
- `scripts/ui/timeline.js` は 988 行
- 自動テストは未整備
- プロダクション URL は `https://xiv-srd.com`

テスト観点の状態:

- `parseFFLogsUrl` は独立しやすい
- `computeRollingDps` はほぼ純粋関数
- `summarizeEvents` と `normalizeErrorMessage` はそのままテストしやすい
- `detectPhases` は `logDebug` 依存がある
- `buildFightPhasesFromFFLogs` は `state.lang` 依存がある

そのため、テスト導入の前に `最小限の抽出` を入れる

## 最優先

### 1. 本番安定化

目的:
公開サイトを安心して触れる状態にする

やること:

- `xiv-srd.com` 上での実機確認手順を固定する
- FFLogs API エラー時の表示を整理する
- 比較失敗時の `サンプルデータ表示` を見直す
- `異なる encounter は比較できません` という文言と実際の挙動を一致させる
- tutorial / analytics / main の導線を整理する

最初の 1 タスク:

- `サンプルデータ表示` を廃止し、明示的なエラー状態に置き換える

別トラックで決めること:

- `encounter mismatch` を警告にするか、比較不可にするか

完了条件:

- 公開サイトで `読み込み -> 戦闘選択 -> プレイヤー取得 -> 比較` が再現確認できる
- 失敗時に誤解のない UI になる

### 2. テスト可能化のための最小抽出

目的:
テストの前に、依存が絡む関数を小さく切り出して壊れにくくする

やること:

- `detectPhases` から `logDebug` 依存を外す
- `buildFightPhasesFromFFLogs` から `state.lang` 依存を外す
- phase / DPS / analytics / proxy error の純粋関数を別ファイルへ寄せる

完了条件:

- テスト対象関数が DOM や `state` に依存せず呼び出せる

### 3. テストの最小導入

目的:
FFLogs スキーマ変化やロジック修正で壊れにくくする

方針:

- テストランナーは `Vitest` を採用する

やること:

- `parseFFLogsUrl`
- フェーズ組み立て処理
- DPS 集計処理
- analytics 集計処理
- エラーメッセージ正規化

上記の純粋関数から先に unit test 化する

完了条件:

- `npm test` で最低限のロジックテストが走る
- 主要な比較ロジックの回帰が検知できる

## 次の優先

### 4. コア UX 改善

目的:
比較ツールとしての使いやすさを上げる

やること:

- 入力中・取得中・失敗時の状態表示を整理する
- 比較結果の shareable state を検討する
- ログ URL、戦闘、プレイヤー選択の復元を入れる
- フェーズ選択とズーム周りの操作感を調整する
- analytics ページへの導線を必要なら限定付きで追加する

### 5. analytics v2

目的:
実運用で使える判断材料を増やす

やること:

- `session_id` 相当の識別子をイベントに追加する
- `reports_loaded` と `comparison_completed` の詳細項目を増やす
- エラー分類を `load_reports / load_players / compare / render` まで揃える
- analytics ダッシュボードに期間切替と上位失敗要因を追加する

完了条件:

- 何が使われていて、どこで失敗しているかをダッシュボードだけで追える

## 中期

### 6. 保守性改善

目的:
今後の機能追加コストを下げる

やること:

- `timeline.js` を描画、フェーズ、DPS、ボス詠唱で分割する
- `fflogs.js` を API 呼び出し、変換、アイコン解決で分割する
- 共通型や定数を寄せる
- デバッグログの責務を整理する

完了条件:

- 1 ファイルが巨大化しすぎず、責務ごとに追える

### 7. 比較機能の拡張

候補:

- wipe 戦闘への対応
- 比較対象の条件絞り込み
- 表示レーンの ON/OFF
- 画像書き出しや共有用出力

## いまは後回し

- FFLogs OAuth を使った非公開ログ対応
- ユーザーアカウント機能
- 大規模な管理画面化
- フレームワーク移行だけを目的にした全面刷新

その次:

- `encounter mismatch` の仕様を決めて UI と実装を一致させる
